Slide 3
System Architecture
4:10 – 3:10
🗣 Say This
  • Walk the diagram left to right: "START board fires a LoRa packet — FINISH board receives it and starts a timer."
  • "When the run ends, the FINISH board POSTs over HTTPS to API Gateway."
  • "A Lambda stores the run in DynamoDB and publishes to SNS — Simple Notification Service — which fans out to a second Lambda."
  • "That Lambda pushes the result to every connected browser via WebSocket. Dashboard updates in under 500ms."
🧠 Know This
  • The LoRa packet is just a run number (START:1) — no timestamp. The finish board starts its own local timer on packet receipt.
  • SNS is the fan-out. Right now one subscriber, but trivial to add logging, SMS, etc.
  • DynamoDB schema: timing-runs table — partition key: date, sort key: timestamp.
Most important slide. Don't rush it. Point at each box as you say its name — helps on a screen share.
Slide 4
Hardware
3:10 – 2:25
🗣 Say This
  • "Heltec LoRa 32 V3 — ESP32 dual-core with an SX1262 LoRa radio built in. $25 each. Right now the trigger is just a button — no soldering yet. Proper hardware trip sensors would need it, but that's on the roadmap."
  • "LoRa is a long-range radio protocol — boards talk directly at 905 MHz. Only the FINISH board needs internet to reach AWS, so a phone hotspot works fine."
  • "My setup gets around 500m — more than enough for any track. The LoRa world record is 830 miles, set with a high-altitude balloon. So we're leaving a little range on the table."
  • "The board even has solar charging input — wild for $25."
🧠 Know This
  • Firmware is C++ (Arduino framework). Interrupt-driven LoRa receive — no polling, just a flag that fires when a packet arrives.
  • FINISH board supports up to 10 concurrent runs in a queue (FIFO) — could stage multiple cars.
  • OLED screens are broken from sun damage — fine for the POC, the dashboard is the wow moment.
If the boards are on your desk, hold them up briefly. Two seconds — much more real than a slide.
Slide 5
Cloud Stack
2:25 – 1:35
🗣 Say This
  • "Everything is in Terraform — all four Lambda functions, both API Gateways, the SNS topic, the database. One terraform apply and the whole stack is up."
  • "Big thing I want to call out: I used import blocks — a TF 1.5+ feature. You declare what you want to import in the .tf file, run terraform plan -generate-config-out, and it writes the resource config for you directly from live state. Completely different from the old CLI import."
  • "The part I found really interesting is how the live push to every browser works — there's no persistent server. When a browser opens the dashboard, API Gateway assigns it a unique connection ID and fires a Lambda that writes it to DynamoDB. That's the live connection registry. When a result comes in, the broadcast Lambda reads that list and pushes to every client. DynamoDB doing double duty — run history and connection registry."
  • (optional if time) "Cleanup is lazy and kind of elegant — if a tab crashes without a clean disconnect, the dead ID just sits in DynamoDB until the next run. When broadcast tries to push to it, the Management API returns a GoneException. Lambda catches it, deletes the stale ID on the spot, and moves on. No separate cleanup job needed."
🧠 Know This
  • Import blocks are idempotent and version-controlled — the old terraform import CLI was a one-shot side effect with no record in code. Import blocks live in .tf files, get reviewed in PRs, can be re-run safely.
  • Two disconnect paths: (1) Clean close → API Gateway fires $disconnect → Lambda deletes the ID immediately. (2) Crash/drop → ID stays in DynamoDB until next broadcast → broadcast Lambda catches GoneException (HTTP 410) from the Management API → deletes stale ID inline. Lazy garbage collection, no sweep job.
  • SNS = Simple Notification Service — the pub/sub fan-out between timing-store and websocket-broadcast.
Lead with Terraform — "all infrastructure is code, version-controlled, no manual console clicks." That lands well with this audience.
Don't get into IAM policy details unless asked. Stay at the "what and why" level.
